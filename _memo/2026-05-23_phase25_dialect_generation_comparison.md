## 実験概要

### 目的

チャットモードで方言回答を生成する際、どちらの方式が方言の精度・自然さの観点で優れているかを定量・定性の両面で比較する。

| パターン | フロー |
|----------|--------|
| **A（直接）** | 標準語の質問 → 方言辞書付きプロンプト → **方言で直接回答** |
| **B（2段階）** | 標準語の質問 → 標準語で回答（Step1）→ 方言に翻訳（Step2） |

### テストフレーズ（5件）

```
- 妙高市に観光に行くことにしたんですが、おすすめの場所はありますか？
- 温泉がいいですね。温泉の特色は何ですか？
- 美味しいものも食べたいです。美味しい郷土料理はありますか？
- 四季それぞれ楽しみがありますよね。どの季節が一番いいですか？
- 小さい子供を3人連れて行くんですけど、家族で遊べる場所はありますか?
```

### 実験条件

| 項目 | 設定 |
|------|------|
| モデル | `gpt-5.4-mini` |
| API | Responses API（`client.responses.create`） |
| `max_output_tokens` | 300（各リクエスト） |
| リクエスト方式 | 独立リクエスト（`previous_response_id` なし） |
| 実行回数 | 2回（本実験）＋ 事前テスト1回 |

## 実験の経緯と改善

### 事前テスト（旧スクリプト）での問題発見

当初のスクリプトはパターンBのStep1プロンプトが汎用的な「親切なアシスタント」であったため、妙高市のコンテキストが欠落した回答が生成された。

**問題の例:**
- Q「美味しい郷土料理は？」→ 北海道・宮城・大阪など全国各地の料理を列挙（妙高固有の情報が出ない）
- Q「家族で遊べる場所は？」→「行く地域を教えてください」（妙高市への訪問であることを把握していない）

この結果、パターンAとBで「答える内容」自体が異なり、方言生成方式の純粋な比較ができない状態になっていた。

### 改善内容（本実験スクリプトへの反映）

**1. 両パターンに共通ペルソナを追加**

```
あなたは妙高市で生まれ育った在住の60代女性のアシスタントです。
観光客や地域の方の質問に答えたり、気軽に会話します。
```

- パターンA: 上記ペルソナ ＋「妙高市の方言で答えてください」＋ 方言辞書
- パターンB Step1: 上記ペルソナ ＋「**方言は使わず、標準語で**簡潔に答えてください」
- パターンB Step2: 翻訳プロンプト ＋ 方言辞書（ペルソナ不要）

**2. パターンB Step2の入力に元の質問を追加（文脈の保持）**

```
【元の質問】{phrase}
【標準語回答】{standard_answer}
```

Step2が「何の質問への回答か」を把握した上で翻訳できるようにした。

## 考察

### 生成方式の技術的な違い

LLMは自己回帰デコーディング（autoregressive decoding）によって1トークンずつ生成する。2つのパターンはタスクの性質が根本的に異なる。

**パターンA: 方言テキストの生成（generation）**

方言指示と辞書を含むシステムプロンプトにより、最初のトークンから方言の確率分布でデコードが始まる。各トークンが方言であり続けるため後続トークンも同じ方言分布に条件付けられ続ける。結果として、語彙・語尾だけでなく、感嘆詞・発話の出だし・文体リズムまで、訓練データの中にある方言話者の発話パターン全体が引き出される。

> **補足: 事前学習済みパラメータと辞書の関係**
>
> プロンプトに渡す辞書はモデルのパラメータを書き換えるものではなく、デコード時のコンテキストとして機能する（ファインチューニングとは異なる）。モデルは各トークン生成時に辞書部分にもアテンションを向けるため、辞書に記載された方言語彙の確率が高くなる（辞書による**条件付け**）。一方、事前学習済みパラメータには方言的な文体・リズム・語用論的パターンが分散表現として埋め込まれており、これは辞書に依存せず独立して機能する。両者は生成中に同時に働く。実際、今回の出力には辞書に明示されていない「じょんのび」「ばか（強調用法）」といった表現が自然に現れており、事前学習済みの方言パターンが辞書と独立して有効であることの実証的な根拠となっている。

**パターンB: 標準語テキストの方言翻訳（translation）**

Step1で完成した標準語テキストが確定する。Step2はその固定されたテキストを入力とする翻訳タスクである。語彙・語尾の対応付けは行われるが、Step1で確定した標準語の統語構造・文体レジスタ・丁寧語の枠組みをStep2が覆すことはできない。

| | パターンA | パターンB |
|--|---------|---------|
| タスク種別 | 方言テキスト生成 | 標準語→方言の翻訳 |
| 方言化の範囲 | 語彙・語尾・文体・発話スタイル全体 | 語彙・語尾の置換が主 |
| 標準語構造の残存 | なし | 統語構造・レジスタが持続 |

### 出力結果への影響

**パターンAで観察された特徴:**

- 感嘆・掛け声が文頭に自然に出現する
  - 例: 「あるてぇ、あるてぇ」「もちろんだよ、おまん」
- 文全体を通じて方言リズムが一貫している
  - 例: 「おらはね、春がいっちゃん好きだわね。…おまんは、どの季節がいっちゃん好きかね？」
- ペルソナ（60代女性・地元民）が語調に反映されている

**パターンBで観察された特徴:**

- Step1の標準語構造が持ち込まれ、語尾だけ変換される
  - 例: 「はい、あるねや」（「はい」は辞書に登録されておらず標準語のまま残る）
  - 例: 「そうですね → そうだねや」（語尾の置換にとどまる）
- 箇条書き・見出し・丁寧語の枠組みがそのまま残る
- 語彙は正確だが、文章全体のリズムが標準語に近い

### ペルソナ追加の効果

両パターンともペルソナに「妙高市出身」を明示したことで、妙高固有の情報（笹寿司、かんずり、いもり池、苗名滝など）が正しく出力された。事前テストとの比較でペルソナによるコンテキスト付与の効果が明確に確認できた。

### トークン打ち切りについて

実験では `max_output_tokens=300` を設定したため、一部の回答が途中で打ち切られた。パターンAは1コールで300トークン以内に収める必要があるが、パターンBはStep1・Step2それぞれ300トークン使えるため合計600トークン分の出力が可能。ただしこれは実験上の設定に起因する制約であり、本番アプリでの `max_output_tokens` 調整で解消できる問題で、パターン選定の本質的な要因とはならない。

## 採用決定

**パターンA（直接方言回答）を採用する。**

| 評価軸 | パターンA | パターンB |
|--------|---------|---------|
| トークン効率 | ◎（少ない） | △（約16%多い） |
| 方言の自然さ | ◎（生成タスクとして一貫） | ○（語彙は正確だが標準語構造が残存） |
| 実装のシンプルさ | ◎（1回のAPIコール） | △（2回のAPIコール・状態管理が必要） |
| 妙高固有知識 | ◎（ペルソナで解決） | ◎（ペルソナで解決） |
| 回答のわかりやすさ | ○ | ◎（標準語回答を中間出力として参照可能） |

採用理由:
1. **方言の自然さが高い** — 生成タスクとして最初のトークンから方言分布でデコードが進むため、翻訳タスクでは引き出しにくい発話スタイル全体のパターンが出力される
2. **トークン効率が優れている** — 本プロジェクトはトークン消費の最小化が必須要件であり、約16%・平均3,100トークンの削減は無視できない
3. **実装がシンプル** — 単一APIコールで完結し、フェーズ3のアプリ実装が複雑にならない
